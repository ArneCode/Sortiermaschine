\hypertarget{index_autotoc_md4}{}\doxysubsubsection{Warum werden Calls im Heap gespeichert?}\label{index_autotoc_md4}
Calls (\mbox{\hyperlink{classCallHandler_a70bf3c03de1953c55fc468b73c77e0f2}{Call\+Handler\+::call\+Ptrs}}) werden mithilfe des {\ttfamily new} Keywords im \href{https://www.geeksforgeeks.org/stack-vs-heap-memory-allocation/}{\texttt{ Heap}} gespeichert um sie weiter benutzen zu können nachdem die Funktion in der sie instanziert wurden abgeschlossen ist. Sie müssen deshalb aber auch manuell mit \mbox{\hyperlink{classCallHandler_a49ab239f771b84300df21bd426788c3e}{Call\+Handler\+::delete\+Calls}} gelöscht werden. \hypertarget{index_autotoc_md5}{}\doxysubsubsection{Warum verschiedene Callable Klassen?}\label{index_autotoc_md5}
Es wäre möglich gewesen statt mehrerer \mbox{\hyperlink{classCallable}{Callable}} Klassen einfach eine zu benutzen und dann die Art des Calls in einer Variable zu speichern. Der Nachteil dieser Methode wäre, dass bei jeder Funktion die etwas mit der Klasse zu tun hat (\mbox{\hyperlink{classCallable_a0e146c329fbdf8720a9541f60fe910e9}{Callable\+::run}}, \mbox{\hyperlink{classCallable_af0778d71c6196b6ba0215cce13f8c7dd}{Callable\+::is\+Done}}, etc.) überprüft werden müsste, was die Art des Calls ist. Das würde zu einer schlechteren Lesbarkeit des Codes führen. Das war in einer älteren Version der Fall.

\hypertarget{index_autotoc_md6}{}\doxysubsection{Optimierungsideen}\label{index_autotoc_md6}
Beim Ausführen des Programms mangelte es manchmal an Speicherplatz. Um dieses Problem zu umgehen hätte man statt der \href{https://www.arduino.cc/reference/de/language/variables/data-types/stringobject/}{\texttt{ Arduino String Klasse}} auch C-\/\+Strings also char \mbox{[}\mbox{]} benutzen können. Das hätte den Code leider aber auch an ein Paar stellen komplizierter gemacht. Ein anderer Fehler der sehr häufig auftrat war, dass mit Speicheradressen falsch umgegangen wurde und dadurch das Programm ohne jegliche Fehlermeldung abstürzte, oft sogar an völlig anderen Stellen. Eine mögliche Lösung wäre es hier eine Programmiersprache zu benutzen, die Speicherplatzfehler beim kompilieren, das heißt beim umwandeln des Programmcodes in Maschinencode aufdeckt. Eine Möglichkeit dafür wäre diese \href{https://github.com/Rahix/avr-hal}{\texttt{ Arduino Bibliothek}} für Rust (\href{https://creativcoder.dev/rust-on-arduino-uno}{\texttt{ Beispiel}}). Solche Bibliotheken sind aber auch weniger Dokumentiert und haben weniger Features als die Standard C++ Arduino Bibliothek und deshalb alles in allem für ein kleines Schulprojekt keine gute Option.
